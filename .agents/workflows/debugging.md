# 🐛 Systematic Debugging — 체계적 디버깅

랜덤 수정은 시간을 낭비하고 새 버그를 만든다. 빠른 패치는 근본 문제를 가린다.

## 핵심 원칙
> **근본 원인을 찾기 전에 절대 수정을 시도하지 않는다. 증상 수정은 실패다.**

## 철칙
```
근본 원인 조사 없이 수정 금지
```
Phase 1을 완료하지 않았다면, 수정을 제안할 수 없다.

## 언제 사용하나
**모든** 기술적 이슈:
- 테스트 실패, 프로덕션 버그, 예상치 못한 동작
- 성능 문제, 빌드 실패, 통합 이슈

**특히 이럴 때:**
- 시간 압박 (급할수록 추측이 유혹적)
- "간단한 수정"이 명확해 보일 때
- 이미 여러 수정을 시도했을 때
- 이전 수정이 작동하지 않았을 때

## 4단계 프로세스

### Phase 1: 근본 원인 조사 🔍
**이 단계를 완료하기 전에 코드를 수정하지 않는다**

1. 증상 수집 — 정확히 무엇이, 언제, 어떻게 발생하는가
2. 재현 확인 — 일관되게 재현할 수 있는가
3. 관련 코드 읽기 — 변경된 코드, 의존하는 코드
4. 로그/에러 메시지 분석

### Phase 2: 패턴 분석 🔎
1. 이 버그가 코드베이스의 더 넓은 패턴의 일부인가?
2. 비슷한 코드에서 같은 문제가 있는가?
3. 최근 변경과의 관계는?

### Phase 3: 가설 & 테스트 🧪
1. 가설 수립 — "X 때문에 Y가 발생한다"
2. 가설을 **검증 가능한 방법**으로 테스트
3. 가설이 맞으면 → Phase 4
4. 가설이 틀리면 → 새 가설

**루프 탈출 가드 (OpenClaw loop-detection 패턴):**
- 같은 수정 시도가 **3회 연속 실패** → "이 접근이 틀렸다" 선언 후 완전히 다른 가설로 전환
- **3가지 가설 모두 실패** → 사용자에게 에스컬레이션 (무한 반복 방지)
- ⚠️ 절대로 같은 수정을 반복하지 않는다

### Phase 4: 구현 ✅
1. 근본 원인을 수정하는 최소 변경 작성
2. 수정이 증상을 해결하는지 확인
3. 회귀 테스트 추가
4. 다른 테스트가 여전히 통과하는지 확인

**커밋 전 검증 (OpenClaw PR truthfulness 패턴):**
커밋 전 아래 4항목 확인 (하나라도 실패 시 커밋 불가):
1. 증상 기술 — 커밋 메시지에 "무엇이 어떻게 잘못되었는지" 명시
2. 근본 원인 특정 — 어떤 파일의 어떤 로직이 문제였는지 확인
3. 수정 범위 일치 — 수정한 코드가 원인 코드 경로에 해당하는지
4. 테스트 통과 — 수정 후 관련 테스트가 통과하는지 (테스트 없으면 수동 검증 기록)

## 위험 신호 — 멈추고 프로세스를 따르라
| 생각 | 현실 |
|------|------|
| "원인은 확실히 X일 거야" | 추측이다. 증거를 찾아라 |
| "이 한 줄만 바꾸면 될 거야" | 왜 그 줄이 문제인지 증명했나? |
| "빨리 고치고 싶다" | 체계적으로 하는 게 실제로 더 빠르다 |
| "이미 3번째 시도인데" | 근본 원인을 찾았나? 아니면 증상만 건드리고 있나? |
