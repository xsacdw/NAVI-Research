# ⚡ Superpowers 워크플로우

이 워크플로우는 obra/superpowers 프레임워크의 개발 방법론을 Antigravity IDE에 적용합니다.
**모든 개발 작업**에 이 프로세스를 따릅니다. "너무 간단한" 프로젝트는 없습니다.

## ⛔ HARD GATES — 절대 건너뛸 수 없음

아래 게이트를 위반하면 **즉시 작업을 중단**하고 빠진 단계를 수행한다.
"빠르게 진행"이나 "나중에 하면 된다"는 이유로 건너뛰기 불가.

### GATE 1: 설계 먼저
- 설계 문서를 `docs/specs/YYYY-MM-DD-<topic>-design.md`에 **저장하고 커밋**한다
- 사용자가 승인할 때까지 프로덕션 코드 작성 금지
- ✅ 체크: `ls docs/specs/*design.md` 로 파일 존재 확인

### GATE 2: 테스트 먼저 (TDD)
- 프로덕션 코드를 작성하기 **전에** 실패하는 테스트를 먼저 작성
- 테스트 없이 작성된 코드는 삭제하고 다시 시작
- 워크플로우 md 수정(비코드)에는 TDD 면제
- ✅ 체크: `pytest tests/ -v` 실행 가능해야 함

### GATE 3: Task마다 커밋
- 각 Task 완료 시 반드시 `git add && git commit`
- 커밋 메시지 형식: `feat|fix|docs|test: 한 줄 설명`
- 여러 Task를 묶어서 한 번에 커밋하지 않는다
- ✅ 체크: `git log --oneline -5` 로 최근 커밋 확인

## 전체 흐름

1. **브레인스토밍** (`/brainstorming` 참조)
2. **계획 작성** (`/writing-plans` 참조)
3. **구현** (`/tdd` + `/executing-plans` 참조)
4. **디버깅** (필요 시 `/debugging` 참조)
5. **리뷰 & 마무리**

## Phase 전환 시 자체 점검

각 Phase 전환 시 아래 체크리스트를 **반드시** 출력한다:

```
🔍 슈퍼파워즈 게이트 점검:
□ GATE 1: 설계 문서 docs/specs/에 저장했는가?
□ GATE 2: 테스트를 먼저 작성했는가? (코드 변경 시)
□ GATE 3: 이전 Task를 커밋했는가?
→ 하나라도 □이면 진행 불가. 먼저 수행한다.
```

## 핵심 원칙

- **설계 먼저, 코드 나중:** 설계 승인 전에 코드를 절대 작성하지 않는다
- **한 번에 한 질문:** 사용자를 여러 질문으로 압도하지 않는다
- **YAGNI:** 요청하지 않은 기능은 만들지 않는다
- **TDD 필수:** 테스트 없이 프로덕션 코드를 작성하지 않는다
- **작은 단위:** 각 태스크는 2~5분 크기로 분할한다
- **빈번한 커밋:** Task마다 커밋, 한 번에 몰아서 금지
