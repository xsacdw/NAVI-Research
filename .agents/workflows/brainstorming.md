# 🧠 Brainstorming — 아이디어를 설계로

자연스러운 대화를 통해 아이디어를 완전한 설계와 스펙으로 만듭니다.

## ⛔ HARD GATE
설계를 제시하고 사용자가 승인할 때까지, 어떤 구현 작업도 하지 않는다.
코드 작성, 프로젝트 생성, 스캐폴딩 — 모두 금지.
이것은 "간단해 보이는" 프로젝트에도 예외 없이 적용된다.

## 체크리스트 (순서대로 수행)

1. **프로젝트 컨텍스트 탐색** — 파일, 문서, 최근 커밋 확인
2. **명확화 질문** — 한 번에 한 질문씩. 가능하면 객관식으로
3. **2~3가지 접근법 제안** — 트레이드오프 설명 + 추천안 제시
4. **설계 제시** — 섹션별로 나눠서 보여주고, 각 섹션마다 승인 확인
5. **설계 문서 작성** — `docs/specs/YYYY-MM-DD-<topic>-design.md`에 저장 및 커밋
6. **사용자 최종 리뷰** — 진행 전 사용자가 스펙 파일을 확인하도록 요청
7. **구현 전환** — `/writing-plans` 워크플로우로 이동

## 설계 프로세스 상세

### 아이디어 이해
- 먼저 현재 프로젝트 상태를 확인한다 (파일, 문서, 커밋)
- 스코프 평가: 요청이 여러 독립 서브시스템을 포함하면 먼저 분해를 제안
- 한 번에 한 질문씩 — 한 메시지에 여러 질문 금지
- 목적, 제약 조건, 성공 기준에 집중

### 접근법 탐색
- 2~3가지 접근법 + 트레이드오프 제시
- 추천안을 먼저 제시하고 이유 설명
- 대화체로 자연스럽게

### 설계 제시
- 복잡도에 맞게 섹션 크기 조절 (간단하면 몇 문장, 복잡하면 200~300 단어)
- 각 섹션 후 "여기까지 괜찮으신가요?" 확인
- 아키텍처, 컴포넌트, 데이터 흐름, 에러 처리, 테스팅 다루기

### 격리와 명확성을 위한 설계
- 하나의 명확한 목적을 가진 작은 단위로 분해
- 각 단위가 잘 정의된 인터페이스로 통신
- 독립적으로 이해하고 테스트 가능해야 함

### 기존 코드베이스에서 작업
- 변경을 제안하기 전에 현재 구조를 탐색
- 기존 패턴을 따름
- 관련 없는 리팩토링은 하지 않음

## 핵심 원칙
- **한 번에 한 질문** — 여러 질문으로 압도하지 않기
- **객관식 선호** — 가능하면 선택지 제시
- **YAGNI** — 불필요한 기능 제거
- **대안 탐색** — 항상 2~3가지 접근법
- **점진적 검증** — 섹션별 승인 후 진행
